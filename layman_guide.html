Q1–Q50

Project Overview & End-to-End Workflow

I built an AI-powered Anti-Money Laundering (AML) Detection System. It takes raw bank transaction data (CSV/Excel files), runs an Isolation Forest ML model to detect suspicious accounts, and then uses a RAG (Retrieval-Augmented Generation) pipeline to automatically generate legally compliant Suspicious Activity Reports (SARs) by referencing PMLA 2002 law documents. The frontend is React and backend is Python Django REST API.

In Indian banks, compliance officers manually review thousands of transactions every day looking for money laundering. This is slow, expensive, and error-prone. My system automates this — it mathematically identifies the top suspicious accounts in seconds and generates a draft legal report (SAR) in under 2 seconds using an LLM. What used to take 3-4 hours of human work now happens in seconds.

REST (Representational State Transfer) is a way for two systems (frontend and backend) to talk using standard HTTP methods. In my project:

  • POST /api/upload/ — uploads CSV file for analysis
  • GET /api/alerts/ — retrieves all flagged suspicious accounts
  • GET /api/alerts/stats/ — returns dashboard summary data (charts, counts)
  • POST /api/alerts/{id}/generate-sar/ — triggers LLM to generate a SAR report
  • GET /api/tasks/{id}/ — checks background processing status

The frontend sends JSON, the backend returns JSON. They are completely decoupled — I could build a mobile app using the same API.

Four models in models.py:

  • Account — Represents a bank account (account_id, name, total_transactions, risk_history as JSONField).
  • Alert — A suspicious finding (risk_score, type like "Structuring", priority as "Critical"/"High", linked to Account via ForeignKey).
  • Transaction — Individual bank transactions (amount, date_time, type, related_account, flag).
  • ProcessingTask — Tracks background processing status (task_id, status: Pending → Processing → Completed, progress percentage).

If the ML analysis ran synchronously inside the API request, the HTTP connection would hang for 30-60 seconds while analyzing 50,000 rows. The browser would likely timeout. By spawning threading.Thread(target=background_process), the API returns HTTP 202 instantly, and the heavy computation runs in the background. The frontend polls for progress every 2 seconds.

I wrote a column mapper in _map_columns() inside risk_engine.py. It has a dictionary like:

'account_id': ['account_id', 'account', 'acc no', 'customer id', 'id']
'amount': ['amount', 'transaction amount', 'value', 'debit', 'credit']

It loops through the uploaded columns and matches them case-insensitively. So whether a bank sends "Acc No" or "Customer ID", the system maps it to our standard account_id. If required columns are missing, it raises a clear ValueError.

The clean_amount() function in _prepare_data() handles all edge cases:

def clean_amount(val):
    if pd.isna(val): return 0.0
    if isinstance(val, (int, float)): return float(val)
    return float(str(val).replace('₹', '').replace(',', '').replace('$', ''))

It strips currency symbols (₹, $), removes commas (49,500 → 49500), handles NaN as 0.0, and returns a clean float for mathematical operations.

In _prepare_data():

  • If both date and time columns exist: concatenate them and parse: pd.to_datetime(dates + ' ' + times).

JWT (JSON Web Token) is a stateless authentication method. When a user logs in, the server generates a signed token containing the user's ID. The frontend sends this token in every request header: Authorization: Bearer . The server verifies the mathematical signature without hitting the database. I used djangorestframework-simplejwt which generates both an Access Token (short-lived, 5 min) and a Refresh Token (long-lived, 24 hours).

Q51–Q100

Machine Learning & Data Engineering

I used Isolation Forest from Scikit-Learn. It's an unsupervised anomaly detection algorithm — perfect for AML because we don't have labeled data (we don't know which accounts are actually criminal). It automatically finds accounts whose behavior is mathematically "weird" compared to the majority.

It means the model doesn't need pre-labeled training data like is_fraud = True/False. In real banking, labels are extremely rare — less than 0.1% of flagged transactions turn out to be actual fraud. Supervised models would starve for positive examples. Isolation Forest works by finding outliers without labels.

Think of it like a game of "20 Questions." The algorithm randomly asks questions about the data (e.g., "Is total_volume > 500,000?"). Normal accounts take many questions to isolate because they're all similar. Suspicious accounts can be isolated in just 2-3 questions because their values are drastically different. Fewer questions to isolate = more anomalous.

Each feature captures a specific money laundering typology. Together they form a 4-dimensional behavioral profile of each bank account.

stats = df.groupby('account_id')['amount'].agg(['sum', 'count'])

This groups all transactions by account_id, then calculates the sum (total money moved) and count (number of transactions) for each account in a single vectorized operation.

df['is_structuring'] = (df['amount'] >= 45000) & (df['amount'] < 50000)
structuring = df.groupby('account_id')['is_structuring'].sum()

Line 1 creates a True/False column — True if the amount is in the ₹45,000-₹49,999 "danger zone" (just below the ₹50,000 CTR threshold). Line 2 counts the Trues per account. If an account has 3+ transactions in this range, it's likely deliberately splitting a large deposit to avoid reporting.

df = df.sort_values(['account_id', 'datetime'])          # Step 1: Sort by time
df['prev_amount'] = df.groupby('account_id')['amount'].shift(1)  # Step 2: Get previous row's amount
df['prev_type'] = df.groupby('account_id')['type'].shift(1)      # Step 3: Get previous row's type
df['time_diff'] = df.groupby('account_id')['datetime'].diff().dt.total_seconds()  # Step 4: Time gap in seconds

df['is_mule'] = (df['type'] == 'Withdrawal') &           # Current is Withdrawal
                (df['prev_type'] == 'Deposit') &          # Previous was Deposit
                (df['time_diff'] <= 86400) &              # Within 24 hours (86400 seconds)
                (df['amount'] >= 0.95 * df['prev_amount']) &  # Amount matches within 5%
                (df['amount'] <= 1.05 * df['prev_amount'])
.shift(1) moves every value in a column down by one row. After sorting by ['account_id', 'datetime'], each row's .shift(1) value is the transaction that happened just before it for the same account. Without sorting first, .shift(1) could bring a value from a completely different account or date — giving meaningless results.

Real money mules keep a 2-5% commission. If someone deposits ₹50,000 and withdraws exactly ₹47,500 (keeping ₹2,500 as their fee), a strict equality check would miss it. The ±5% window catches both exact and near-matching amounts.

round_trips = df[df['related_account'].notna()].groupby(['account_id', 'related_account']).size()
round_trip_counts = round_trips[round_trips > 1].groupby('account_id').count()

It counts how many transactions exist between each (account, counterparty) pair. If Account A interacts with Account B more than once, that's a potential circular flow. The second groupby counts how many such suspicious relationships each account has.

X = features_df[['total_volume', 'structuring_count', 'mule_score', 'round_trip_count']].values

This creates a 2D NumPy array where each row is one account and each column is one feature:

Account_A: [2,450,000,  5,  3,  2]  ← suspicious
Account_B: [   45,000,  0,  0,  0]  ← normal
clf = IsolationForest(random_state=42, contamination=0.1)
  • random_state=42 — Fixed seed for reproducible results (same output every run).
  • contamination=0.1 — Tells the model to expect ~10% of accounts to be suspicious. This sets the decision boundary.

Our test dataset has roughly 10% suspicious accounts (e.g., 1000_rows_100_suspicious.csv = exactly 10%). In real banking, fraud rate is 0.1-1%, so we would lower this to 0.01. Setting it too high creates too many false positives; too low misses real fraud.

scores = clf.decision_function(X)  # Returns raw scores (-0.5 to +0.5)
normalized_scores = (1 - ((scores - min_score) / (max_score - min_score))) * 100

This is Min-Max Normalization. The most negative score (most anomalous) maps to 100, and the most positive score (most normal) maps to 0. The 1 - (...) inverts the scale so higher = more suspicious.

if row['total_volume'] > 1000000: patterns.append('High Volume')
if row['structuring_count'] >= 2: patterns.append('Structuring')
if row['mule_score'] > 0: patterns.append('Money Mule')
if row['round_trip_count'] > 0: patterns.append('Round Trip')
if risk_score > 75 and not patterns: patterns.append('Anomalous Behavior')

This is a rule-based post-processor on top of the ML score. It gives each alert a human-readable explanation. The last rule ("Anomalous Behavior") is a catch-all for accounts that are mathematically strange but don't match any specific typology.

It appears when the ML score is high (>75) but none of the 4 specific rules triggered. This means the combination of features is unusual — for example, moderate volume + moderate structuring + moderate mule activity — none individually alarming, but together statistically rare.

joblib is optimized for large NumPy arrays inside Scikit-Learn models. The model is trained once on a baseline dataset and reused for all future uploads.

Retraining on every upload would change the "definition of normal" with each file. The model should be trained once on a large representative baseline, so all future uploads are scored against the same consistent standard. This ensures auditable, reproducible results.

Vectorization applies an operation to an entire column at once using optimized C code, instead of Python's slow row-by-row loop.

Example: df['amount'] * 1.05 multiplies ALL 100,000 values in a single CPU instruction. A Python for loop doing the same takes 100,000 individual function calls. Vectorized code can be 100x to 1000x faster.

It splits the DataFrame into groups based on a column value. Example: df.groupby('account_id') creates separate mini-DataFrames for each unique account. Then .sum() or .count() operates on each group independently and returns one result per group.

When concatenating the 4 feature series with pd.concat([...], axis=1), some accounts might have no round-trip or mule activity. Those cells become NaN. Scikit-Learn crashes on NaN values. .fillna(0) replaces missing values with 0, meaning "no suspicious activity detected for this feature."

It strips away all Pandas metadata (column names, index labels) and returns a raw NumPy 2D array. Scikit-Learn's .fit() and .decision_function() require pure NumPy arrays, not DataFrames.

It applies multiple aggregation functions to the same grouped column simultaneously:

stats = df.groupby('account_id')['amount'].agg(['sum', 'count'])

Returns a DataFrame with two columns: the total amount (sum) and the number of transactions (count) per account — in a single pass over the data.

.diff() — Subtracts each row from the previous row in the same group, giving a timedelta (time difference).
  • .dt.total_seconds() — Converts that timedelta to a raw float number of seconds.

Result: the exact time gap (in seconds) between consecutive transactions for each account.

.fit(X) — Builds the 100 random trees using the training data. This is the "learning" step.
  • .decision_function(X) — Drops each account through the pre-built trees and measures how quickly it gets isolated. This is the "scoring" step.

RAG Pipeline, LLM & Embeddings

RAG is a technique where instead of relying solely on the LLM's memory, we first retrieve relevant facts from a private database, then augment the prompt with those facts, so the LLM generates a factually grounded answer. In my project, the "facts" are paragraphs from PMLA 2002 law documents stored in a FAISS vector database.

Two critical reasons:

  • Hallucination — If I ask Llama-3.3 "quote PMLA Section 12", it might invent fake section numbers. RAG provides the actual text.
  • Private Data — The LLM has never seen our specific PMLA PDFs or RBI circulars. RAG feeds it the exact legal paragraphs it needs.

It's a mathematical representation of text as an array of numbers. The sentence "Money laundering under PMLA" becomes a 384-dimensional float array like [0.12, -0.45, 0.99, ...]. Semantically similar sentences have similar arrays — their vectors "point in the same direction" in 384-dimensional space.

all-MiniLM-L6-v2 from HuggingFace. It's a small (22 MB), fast, and free model that converts text into 384-dimensional vectors. It runs 100% locally on CPU — no API calls, no internet needed, no cost, and complete data privacy.

FAISS (Facebook AI Similarity Search) is an open-source C++ library by Meta. It stores vectors and finds the nearest neighbors extremely fast — even among millions of vectors. In my project, it stores the embedded PMLA law chunks and retrieves the most relevant ones for each SAR query.

It measures the angle between two vectors. If two vectors point in the same direction (angle = 0°), cosine similarity = 1.0 (identical meaning). If they're perpendicular (90°), it's 0.0 (unrelated). FAISS uses this to find which law chunks are most semantically similar to the query.

It splits text at natural boundaries using a priority list: first tries double newlines (\n\n — paragraphs), then single newlines, then spaces, then individual characters. This ensures legal clauses aren't cut mid-sentence. chunk_size=1000 limits each chunk to ~1000 characters.

If a key legal phrase spans the boundary between chunk 1 and chunk 2, without overlap it would be split across both chunks and neither would contain the complete phrase. The 100-character overlap ensures the phrase appears complete in at least one chunk.

Temperature controls the randomness of the output:

  • temperature=0.0 — Completely deterministic, always picks the highest-probability token
  • temperature=0.1 (our setting) — Almost deterministic but with slight variation. Perfect for legal documents where accuracy is critical but slight phrasing variation is acceptable.

The maximum number of tokens (input + output) an LLM can process in one call. Llama-3.3 has 128K tokens. Our full prompt (system + laws + evidence + template) uses about 3,000-5,000 tokens — well within limits. If we retrieved k=50 chunks instead of k=5, we'd risk overloading the context.

Financial Intelligence Unit - India, under the Ministry of Finance. Banks must file Suspicious Transaction Reports (STRs) with FIU-IND within 7 days of detection. Our SAR generator produces the draft of this legally mandated document.

Under PMLA, banks must automatically report any cash transaction above ₹10,00,000 (or series of transactions aggregating to ₹10,00,000 in a month) to FIU-IND. Criminals avoid CTRs by "structuring" — breaking deposits into amounts below the threshold.

The ₹45,000-₹49,999 range in extract_features_vectorized() targets deposits deliberately kept just below ₹50,000. If an account has 2+ transactions in this range, the structuring_count >= 2 rule flags it. The ML model independently scores the mathematical unusualness of this pattern.

A person (often unwitting) who receives money in their bank account and immediately transfers it elsewhere, acting as a transit point to obscure the money trail. Our mule_score detects accounts where a Withdrawal closely follows a Deposit of a similar amount within 24 hours.

A shell company is a registered entity with no real business operations — used purely as a financial conduit. Money flows: Criminal → Shell A → Shell B → Criminal. Our round_trip_count detects when the same related_account appears repeatedly, which signals circular money movement through shell entities.

KYC is the identity verification process during account opening — collecting ID proof, address proof, and understanding the customer's expected transaction behavior. Our system focuses on the post-KYC monitoring — continuous analysis of actual transactions to detect suspicious behavior after the customer is already onboarded.

EDD is a heightened investigation triggered for high-risk accounts. When our SAR recommendation says "Consider Enhanced Due Diligence," it means the compliance team should collect additional documentation: beneficial ownership proof, source of funds declarations, and business activity justification.

FATF (Financial Action Task Force) is the global standard-setter for AML. Its 40 Recommendations define what every country's AML laws must cover. India's PMLA 2002 was drafted to comply with FATF standards. Our LLM prompt references FATF as a fallback when specific PMLA sections don't apply.

The incoming CSV data often contains currency symbols and commas: "₹49,500". If I used DecimalField, Django would crash trying to validate this string. CharField preserves the raw value. The volume sum is calculated at query time by casting to float.

Sum(Cast('amount', FloatField()))

Since amount is a CharField in the database, we can't use Sum() directly. Cast('amount', FloatField()) tells the database to convert the string to a number before summing. If this fails (amounts have ₹ symbols), a Python fallback handles the conversion.
